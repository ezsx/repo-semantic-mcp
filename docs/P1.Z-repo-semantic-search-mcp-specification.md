## 0) Контекст и цель

В репозитории уже есть сильный набор deterministic-инструментов:
- `code-index` для shortlist файлов;
- `serena` для symbol/references;
- `ast-grep` для structural поиска и refactor;
- `ripgrep` как fallback.

Этого достаточно для точечной инженерной работы, но недостаточно для двух классов запросов:
1. **Семантический natural-language поиск по смешанному RU/EN репозиторию.**
   Примеры: "где у нас резервируются invite-коды", "кто отвечает за stale policy", "где описан rollout observability".
2. **Cross-domain retrieval между кодом, архитектурой, спецификациями и guide-ами.**
   Нужен один semantic shortlist перед переходом к symbol/AST инструментам.

Цель этой спецификации:
1. Ввести в репозиторий **полноценный repo-owned MCP server** для vector/hybrid поиска.
2. Отделить **векторное хранилище** от MCP-процесса через отдельный `Qdrant` сервис.
3. Встроить semantic retrieval в текущий агентский workflow, не ломая существующий `code-index -> serena -> ast-grep`.
4. Заложить поддерживаемый layout, чтобы semantic MCP стал таким же repo-owned компонентом, как `agent_context/*`.

## 1) Утвержденные решения

1. **Semantic search вводится как отдельный MCP server, а не как ad-hoc script или локальный notebook.**
   Источник: официальный MCP Python SDK поддерживает полноценные MCP servers с tools/resources/prompts и стандартными transport'ами (`stdio`, `SSE`, `Streamable HTTP`).

2. **Векторное хранилище живет в отдельном `Qdrant` сервисе.**
   Local mode `qdrant-client` допускается только для локальной отладки и unit/integration тестов, но не как основной runtime контур.

3. **v1 retrieval строится как hybrid dense + lexical.**
   Dense retrieval обязателен.
   Lexical/sparse сигнал обязателен.
   Cross-encoder/late-interaction rerank разрешён как follow-up, но не блокирует v1.

4. **Модель dense embeddings по умолчанию: `BAAI/bge-m3`.**
   Причина: репозиторий смешивает английский код и русские архитектурные/операционные документы; нужна multilingual retrieval модель без отдельной query instruction.

   **Практическая оговорка rollout:**
   если конкретный backend не поддерживает `bge-m3` в production-like runtime,
   допускается временный deploy profile на другой multilingual TEI-compatible модели,
   пока `bge-m3` path не валидирован end-to-end.

5. **v1 embedding backend проектируется как абстракция с двумя провайдерами:**
   - `fastembed_local` — dev/fallback профиль;
   - `tei_http` — рабочий shared deployment profile для контейнерного runtime.

   **Практическая оговорка rollout:**
   на текущем этапе production-like deploy по умолчанию идёт через `tei_http`, потому что
   это даёт устойчивый cold-start и не привязывает индекс к in-process embedding runtime.

6. **Semantic MCP не заменяет существующий symbolic stack.**
   Нормативный workflow после внедрения:
   - semantic MCP -> shortlist
   - `serena`/`code-index` -> уточнение symbols/files
   - `ast-grep`/edit tools -> изменение кода

7. **Индексируются chunk-level сущности, а не whole-file blobs.**
   Основные чанки:
   - Python symbols (function/class/method/module preamble);
   - Markdown sections;
   - YAML/JSON top-level blocks;
   - fallback text chunks для небольших operational файлов.

8. **Индекс является read-only производным представлением репозитория.**
   MCP server не вносит изменения в код и не хранит никаких write-capabilities.

9. **Индекс логически разделяется на две коллекции: `code` и `docs`.**
   Агент должен уметь явно выбирать code-only или docs-only retrieval path.
   Общий `all` scope разрешён только как удобный union поверх этих двух коллекций.

10. **Markdown-отчёты из `tools/testing/load/reports/**/*.md` индексируются по умолчанию.**
    Текущий объём корпуса мал, а operational ценность таких отчётов уже сейчас заметна.

## 3) Scope / Non-goals

### Входит в scope
- Новый repo-owned MCP server для semantic retrieval.
- Отдельный `Qdrant` runtime сервис.
- Индексация кода и документации репозитория.
- Dense + lexical retrieval.
- Полный rebuild индекса и частичный reindex по paths.
- Incremental update через file watcher с debounce.
- Поддержка `stdio` режима для локальных агентов.
- Поддержка `Streamable HTTP` как optional mode для shared deployment.
- Документация, env contract и install/run guide.

### Не входит в scope v1
- Генерация ответов LLM поверх найденных чанков.
- Автоматическое редактирование репозитория через semantic MCP.
- Индексация бинарников, изображений и артефактов нагрузочных прогонов.
- Multi-repo federated search.
- Распределённая кластеризация Qdrant.
- Обязательный cross-encoder reranking на каждый запрос.
- Замена `serena`/`code-index`/`ast-grep`.

## 5) Архитектура решения

### 5.1 Компоненты

1. **Repo Semantic MCP**
   - Python сервис на официальном MCP SDK;
   - экспонирует tools/resources для semantic retrieval и индексации;
   - сам не хранит индекс в памяти как source-of-truth;
   - ходит в Qdrant и в embedding backend.

2. **Qdrant**
   - отдельный сервис с persistent volume;
   - хранит dense vectors и payload metadata;
   - в v1 также хранит lexical/sparse представление для hybrid retrieval;
   - коллекция versioned и завязана на embedding model/version.

3. **Embedding backend**
   - интерфейс внутри `services/repo_semantic/embeddings/*`;
   - реализации:
     - `FastEmbedProvider`
     - `TeiProvider`
   - provider используется и для indexing, и для query embedding.

4. **Indexer / Watcher**
   - строит чанки из файлов репозитория;
   - считает content hash;
   - upsert/delete в Qdrant;
   - умеет `full rebuild`, `reindex_paths`, `watch mode`.

### 5.2 Transport model

Нормативный transport contract:
- **default:** `stdio` для локальных Codex/Claude сессий;
- **optional:** `Streamable HTTP` для shared daemon режима.

Причина:
- `stdio` проще для локальной интеграции агентов;
- separate Qdrant уже даёт общий persistent index;
- `Streamable HTTP` оставляется для later shared deployment, но не должен блокировать первый rollout.

### 5.3 Интеграция в текущий workflow

После внедрения ожидаемый workflow агента:
1. Natural-language запрос -> semantic MCP (`semantic_search` / `hybrid_search`);
2. shortlist files/chunks/symbols;
3. `serena` / `code-index` для точного открытия symbol scope;
4. `ast-grep` или symbol-based edit для рефакторинга;
5. файлы читаются только точечно.

## 6) Chunking и индексируемый корпус

### 6.1 Include scope

В индекс по умолчанию входят:
- `apps/**`
- `services/**`
- `libs/**`
- `docs/**`
- `deploy/**`
- `scripts/**`
- `tools/**`
- `agent_context/**`
- `tools/testing/load/reports/**/*.md`

### 6.2 Default exclude scope

По умолчанию исключаются:
- `.git/**`
- `.venv/**`, `venv/**`, `__pycache__/**`
- `node_modules/**`
- `artifacts/**`
- `tmp/**`, `temp/**`
- `docs/specifications/telegram-bot/image/**`
- бинарники/архивы/изображения/медиа
- все `.env*`, `*.pem`, `*.key`, `*.crt`, `*.pfx`, `*.kdbx`
- generated docs и другие большие производные артефакты, если они появятся в конкретном репозитории

### 6.3 Chunking rules

#### Python
- top-level function -> отдельный chunk;
- class -> отдельный summary chunk;
- method -> отдельный chunk;
- module preamble/import area -> отдельный lightweight chunk, если там есть значимый operational context;
- metadata должны включать `symbol_path`, `kind`, `language=python`.

#### Markdown
- chunk = heading section (`#`, `##`, `###`);
- слишком длинные секции режутся на подчанки с наследованием heading path;
- metadata включают `heading_path`, `doc_type`.

#### YAML / JSON / Compose
- chunk = top-level key/block или файл целиком, если он мал;
- metadata включают `config_kind`, `top_level_key`.

#### Plain text / shell / env-like docs
- chunk = окно строк разумного размера;
- для небольших operational файлов допускается whole-file chunk, если файл короткий и не generated.

### 6.4 Chunk metadata

Каждый point/chunk обязан иметь payload:
- `repo_root`
- `relative_path`
- `language`
- `chunk_type`
- `symbol_path` или `heading_path`
- `start_line`
- `end_line`
- `content_hash`
- `source_mtime`
- `domain_tags` (например: `flow01`, `telegram_bot`, `observability`)
- `is_generated`
- `embedding_model`
- `embedding_backend`
- `index_schema_version`

## 7) Retrieval design

### 7.1 Почему hybrid, а не pure vector

Для кода pure vector retrieval недостаточен:
- он лучше ловит смысл, но хуже удерживает точные идентификаторы;
- по конфигам и operational именам сильный lexical signal обязателен;
- по mixed RU/EN corpus нужен и semantic, и keyword слой.

Поэтому v1 retrieval должен быть hybrid:
- dense vector retrieval;
- lexical/sparse retrieval;
- объединение результатов с нормализацией score;
- optional rerank для top-N как follow-up.

### 7.2 Dense retrieval

Dense retrieval по умолчанию использует:
- model: `BAAI/bge-m3`
- distance: cosine

Причины выбора:
- multilingual retrieval;
- хорошо подходит под mixed-language repo;
- не требует отдельной query instruction;
- допускает future path к sparse/late-interaction, если это потребуется позже.

### 7.3 Lexical / sparse retrieval

В v1 lexical/sparse слой должен быть проще dense слоя:
- по умолчанию использовать BM25/sparse path, совместимый с Qdrant hybrid pipeline;
- не завязывать v1 на multi-vector/late-interaction embedding pipeline.

Это сознательное упрощение:
- dense уже закрывает semantic gap;
- lexical слой закрывает exact identifiers;
- сложный late-interaction rerank можно добавить позже на top-K без пересборки всей архитектуры.

### 7.4 Reranking

Reranking в v1:
- **не обязателен** на каждом запросе;
- разрешён как optional feature flag для `top_k <= 50`;
- должен включаться только после появления стабильного baseline hybrid retrieval.

Причина:
- rerank повышает качество, но увеличивает latency и operational surface;
- сначала нужен стабильный semantic shortlist, а не максимальная исследовательская сложность.

## 8) MCP contract

### 8.1 Обязательные tools

#### `semantic_search`

Назначение:
- dense-first semantic retrieval по natural-language запросу.

Минимальные аргументы:
- `query: str`
- `top_k: int = 10`
- `path_prefix: str | None = None`
- `chunk_types: list[str] | None = None`

Возвращает:
- список результатов с `score`, `relative_path`, `chunk_type`, `symbol_path/heading_path`, `line_range`, `snippet`.

#### `hybrid_search`

Назначение:
- dense + lexical retrieval для mixed queries.

Минимальные аргументы:
- `query: str`
- `top_k: int = 10`
- `path_prefix: str | None = None`
- `domain_tags: list[str] | None = None`

Возвращает:
- результат того же формата + diagnostic fields:
  - `dense_score`
  - `lexical_score`
  - `final_score`

#### `find_similar_chunk`

Назначение:
- поиск похожих символов/секций по уже известному chunk.

Минимальные аргументы:
- `relative_path: str`
- `symbol_path_or_heading: str`
- `top_k: int = 10`

#### `read_chunk`

Назначение:
- получить полный текст конкретного индексированного chunk по `chunk_id`.

Это нужно, чтобы semantic MCP не заставлял агент читать файл целиком.

#### `index_status`

Возвращает:
- активный collection name;
- embedding model/backend;
- schema version;
- число point/chunk;
- last full build timestamp;
- watcher status;
- index freshness summary.

#### `rebuild_index`

Назначение:
- полный rebuild всего индекса.

Аргументы:
- `full: bool = True`
- `clear_first: bool = False`

#### `reindex_paths`

Назначение:
- частичный reindex списка путей.

Аргументы:
- `paths: list[str]`

### 8.2 Optional tools

Разрешены во второй очереди:
- `search_docs_and_code`
- `search_by_error`
- `search_related_decisions`
- `list_indexed_paths`

### 8.3 Resources

Разрешён minimal set resources:
- `index://status`
- `index://collections`
- `index://config`

Но основной сигнал должен идти через tools, а не через bulk resources.

## 9) Хранилище и схема коллекции

### 9.1 Collection strategy

Коллекция обязана быть versioned.

Рекомендуемый шаблон имени:
`repo_semantic_{embedding_family}_{schema_version}`

Пример:
`repo_semantic_bge_m3_v1`

### 9.2 Point identity

`point_id` должен быть детерминированным и стабильным.

Рекомендуемая формула:
`sha256(relative_path | chunk_locator | content_hash | schema_version)`

Это позволяет:
- делать идемпотентный upsert;
- корректно обновлять изменённые чанки;
- безопасно удалять ушедшие чанки.

### 9.3 Payload indexes

Для performance и filtered retrieval должны быть payload indexes минимум по:
- `relative_path`
- `language`
- `chunk_type`
- `domain_tags`
- `symbol_path`
- `heading_path`
- `content_hash`

## 10) Embedding backend abstraction

### 10.1 Интерфейс

Внутри `services/repo_semantic/embeddings/` вводится общий интерфейс:
- `embed_documents(texts: list[str]) -> list[vector]`
- `embed_query(text: str) -> vector`
- `backend_name() -> str`
- `model_name() -> str`
- `healthcheck()`

### 10.2 `FastEmbedProvider`

Используется как dev/fallback backend.

Причины:
- меньше moving parts;
- быстрая интеграция с Qdrant ecosystem;
- лёгкий CPU/GPU runtime;
- подходит для локальной разработки и первого production-like прототипа.

### 10.3 `TeiProvider`

Используется как основной shared runtime backend при желании:
- вынести inference из MCP процесса;
- использовать отдельный GPU/CPU сервис;
- получить production-friendly metrics/traces на inference контуре.

Практический default profile на 2026-03-08:
- backend: `tei_http`
- model: `intfloat/multilingual-e5-small`

Причина:
- `bge-m3` остаётся target dense model в архитектуре;
- `multilingual-e5-small` сейчас даёт приемлемый cold-start и стабильный container-first runtime.

### 10.4 Нормативное правило

Qdrant и embedding provider не должны быть жёстко слиты.

То есть:
- замена `fastembed_local` на `tei_http` не должна менять MCP contract;
- смена dense model должна приводить к полному rebuild индекса, но не к переписыванию API tools.

## 11) Инкрементальный индексатор и watcher

### 11.1 Full rebuild

Полный rebuild должен:
1. просканировать include scope;
2. отфильтровать denylist/generated/secrets;
3. построить chunk graph;
4. сгенерировать embeddings;
5. upsert в collection;
6. удалить stale point'ы, которых больше нет в репозитории.

### 11.2 Incremental reindex

Incremental режим должен:
- пересчитывать только изменившиеся файлы;
- удалять point'ы удалённых файлов;
- работать по `content_hash`, а не только по `mtime`.

### 11.3 Watch mode

Watcher в v1 разрешён, но с guardrails:
- debounce не меньше `2-5` секунд;
- burst changes батчатся;
- rebuild не запускается параллельно сам с собой;
- статус rebuild виден через `index_status`.

## 12) Security и data hygiene

### 12.1 Read-only правило

Semantic MCP не должен:
- выполнять shell;
- изменять файлы;
- делать сетевые вызовы вне:
  - Qdrant
  - embedding provider

### 12.2 Secret hygiene

Индексатор не должен индексировать:
- `.env*`
- ключи/сертификаты/секреты
- runtime dumps
- plaintext credentials

Если файл матчится denylist, он не попадает ни в chunking, ни в payload metadata.

### 12.3 Prompt/data hygiene

В output tools нужно возвращать:
- короткий snippet;
- path/symbol metadata;
- без чрезмерного bulk payload.

Это нужно, чтобы semantic MCP экономил контекст, а не раздувал его.

## 15) Acceptance criteria

Спецификация считается реализованной, когда выполнены все критерии ниже.

### AC1: Semantic retrieval по mixed-language запросам

Запросы уровня:
- "где описан FLOW-01 контракт"
- "где в телеграм боте активируется invite"
- "где описан rebuild производной архитектурной документации"

возвращают релевантные результаты из кода и документов без ручного указания path.

### AC2: Hybrid retrieval не теряет exact identifiers

Запросы уровня:
- `FLOW01_POLICY_ID`
- `SLOT_CONFIG_MASTER_KEY`
- `device_menu_refresh_failed`

возвращают точные релевантные чанки, а не только семантически похожий шум.

### AC3: Incremental update

После изменения индексируемого файла:
- `reindex_paths` или watcher обновляет результат без полного rebuild;
- старый content hash исчезает из выдачи.

### AC4: Agent-friendly output

Результаты tools:
- компактны;
- содержат path/symbol/line metadata;
- не возвращают огромные whole-file blobs.

### AC5: Read-only безопасность

MCP server не имеет write-capabilities и не индексирует denylisted secret files.

## 18) Итоговое решение

Для репозитория вводится **repo-owned semantic MCP** со следующим нормативным дизайном:
- отдельный MCP server на официальном Python SDK;
- отдельный `Qdrant` сервис как persistent vector store;
- dense embeddings через `bge-m3`;
- backend abstraction `fastembed_local | tei_http`;
- hybrid retrieval как основной search mode;
- semantic MCP используется как shortlist layer перед `serena`/`ast-grep`.

Это решение выбрано как минимально рискованное и одновременно достаточно сильное, чтобы дать
агентам repo-wide semantic retrieval без деградации поддерживаемости и без потери текущего deterministic workflow.
